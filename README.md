HNU_NHS_Vision Git使用说明（25级专属）

本文档为海南大学南海鲨战队（HNU_NHS）视觉组 Git 仓库使用规范，适用于25级所有参与视觉组开发的同学及各组负责人，严格遵循本说明操作，确保代码仓库整洁、协作高效。

仓库地址：https://github.com/xq395/HNU_NHS_Vision.git

核心原则：不直接操作主分支、分支命名规范、提交信息规范、先验收后合并

一、前期准备（首次使用必做）

1. 环境配置

- 安装 Git：从 Git 官方网站（https://git-scm.com/）下载对应系统版本，默认安装即可（安装过程中勾选“Add Git to PATH”，方便终端使用）。

- 安装 VSCode：已安装的同学忽略，未安装的前往官网（https://code.visualstudio.com/）下载，安装后打开 VSCode，安装“Git”“GitHub Pull Requests and Issues”插件（方便可视化操作Git）。

- GitHub 登录：确保已登录个人 GitHub 账号，且已被加入战队仓库协作列表（联系负责人添加）。

2. 仓库相关说明

2.1 目录结构

仓库根目录固定如下，25级同学需根据自己所属组别，在对应目录下开发，禁止跨目录操作（如识别组同学不可修改解算组目录内容）：

HNU_NHS_Vision/
├── detection/      # 识别组（2人）：负责目标检测、装甲板识别等相关开发
├── solve/          # 解算组（4人）：负责坐标解算、PnP算法等相关开发
├── communication/  # 通信组（2人）：负责串口通信、数据传输等相关开发
├── docs/           # 文档目录：存放开发文档、说明文档、规范文档（所有人可查看，负责人维护）
├── .gitignore      # Git忽略文件：过滤无用文件（禁止修改）
├── README.md       # 项目说明文档（禁止随意修改，修改需联系负责人）
└── main.cpp        # 视觉组核心入口文件（修改需经负责人同意）


2.2 分支结构（核心）

仓库分支分为「固定分支」和「个人分支」，所有操作需严格遵循分支规则，禁止违规操作。

固定分支（远程仓库已创建，禁止直接修改、推送）

- main：主分支，存放稳定、可发布的代码，仅负责人可申请合并，绝对禁止任何同学直接在main分支开发、推送代码。

- dev/detection：识别组开发总分支，由识别组负责人维护，仅接收本组组员的代码合并，禁止组员直接推送。

- dev/solve：解算组开发总分支，由解算组负责人维护，仅接收本组组员的代码合并，禁止组员直接推送。

- dev/communication：通信组开发总分支，由通信组负责人维护，仅接收本组组员的代码合并，禁止组员直接推送。

个人分支（25级同学专属，自行创建、维护）

命名规范（强制统一，否则负责人有权拒绝验收）：

feature/组别-姓名-功能描述

举例：

- 识别组同学：feature/detection-张三-装甲板识别算法

- 解算组同学：feature/solve-李四-PnP坐标解算

- 通信组同学：feature/communication-王五-串口数据接收

说明：姓名用真实姓名（或常用昵称，确保负责人能识别），功能描述简洁明了，避免模糊（如“feature/detection-张三-功能开发”此类命名无效）。

二、25级同学操作流程（核心流程，必严格遵循）

1. 首次克隆仓库（仅做一次）

打开 VSCode，按以下步骤操作：

1. 点击 VSCode 左上角「文件」→「克隆仓库」。

2. 在弹出的输入框中，粘贴仓库地址：https://github.com/xq395/HNU_NHS_Vision.git

3. 选择本地保存路径（建议路径中不要有中文、空格，如：D:\HNU_RM\25_ws\HNU_NHS_Vision）

4. 点击「克隆」，等待克隆完成后，VSCode 会自动打开克隆后的仓库

终端命令备用（若可视化操作失败，打开 VSCode 终端，输入以下命令）：

# 切换到本地保存路径（示例路径，需替换为自己的路径）
cd D:\HNU_RM\25级相关资料
# 克隆仓库
git clone https://github.com/xq395/HNU_NHS_Vision.git

2. 切换到本组开发分支（每次开发前必做）

克隆完成后，默认处于 main 分支，需先切换到自己所属组的开发分支，并拉取最新代码（避免代码冲突）

可视化操作：

1. 点击 VSCode 左下角的分支名称（默认显示 main）

2. 在弹出的分支列表中，找到自己组的开发分支（识别组选 dev/detection，解算组选 dev/solve，通信组选 dev/communication）

3. 点击「检出到」，完成分支切换

4. 切换完成后，点击左下角「同步」按钮（两个箭头循环图标），拉取最新代码

终端命令备用：

# 识别组同学
git checkout dev/detection
git pull

# 解算组同学
git checkout dev/solve
git pull

# 通信组同学
git checkout dev/communication
git pull

3. 创建个人开发分支（每次开发新功能必做）

切换到本组开发分支并拉取最新代码后，创建自己的个人分支，所有开发工作均在个人分支上进行。

可视化操作：

1. 点击 VSCode 左下角的分支名称（当前显示为 dev/xxx，xxx 为自己组的分支名）。

2. 点击「创建新分支」，输入个人分支名称（严格遵循命名规范）。

3. 选择「从当前分支（dev/xxx）创建」，完成个人分支创建（自动切换到个人分支）。

终端命令备用：

# 替换分支名为自己的个人分支（示例为识别组同学）
git checkout -b feature/detection-张三-装甲板识别算法

4. 日常开发与提交（多次操作，小步快跑）

在个人分支上进行代码开发，遵循“小步快跑、频繁提交”的原则，每完成一个小功能（如一个函数、一个接口,而不是一行代码更改），就提交一次代码，避免一次性提交大量代码（便于回滚和问题排查）。

1. 开发完成一小部分功能后，检查自己修改的文件（VSCode 左侧「源代码管理」面板会显示修改的文件）。

2. 暂存文件：点击每个修改文件旁边的「+」号，或点击「更改」旁边的「+」号（全选），将文件暂存（暂存表示告诉 Git，这些文件需要被提交）。

3. 填写提交信息：在「消息」输入框中，填写规范的提交信息（强制遵循格式，否则负责人有权拒绝验收）。

提交信息规范（格式：类型: 功能描述，简洁明了，不超过50字）：

- feat：新功能开发（如：feat: 完成装甲板识别基础函数）

- fix：修复bug（如：fix: 修复装甲板识别误判问题）

- docs：文档修改（如：docs: 补充识别模块开发文档）

- style：代码格式调整（不影响功能，如：style: 调整代码缩进）

示例：feat: 完成PnP坐标解算核心逻辑、fix: 修复串口通信数据丢包问题

1. 提交代码：点击「消息」输入框右侧的「✅」按钮，或按 Ctrl+Enter，完成本地提交。

2. 推送代码到远程：提交完成后，点击 VSCode 左下角的「同步」按钮，或点击「源代码管理」面板中的「推送」按钮，将本地个人分支推送到远程仓库（首次推送会提示“推送分支到远程”，点击确认即可）。

终端命令备用：

# 暂存所有修改文件
git add .
# 提交代码（替换提交信息为自己的）
git commit -m "feat: 完成装甲板识别基础函数"
# 推送个人分支到远程（首次推送需加 -u，后续无需）
git push -u origin 个人分支名
# 后续推送（无需加 -u）
git push

5. 提交验收申请（功能开发完成后）

当个人分支上的功能开发完成、测试通过后，需提交验收申请，由本组负责人验收代码，步骤如下：

1. 打开浏览器，访问仓库地址：https://github.com/xq395/HNU_NHS_Vision.git。

2. 点击页面上方的「Pull requests」（PR）→「New pull request」。

3. 选择源分支和目标分支：
        

  - 源分支（Compare）：选择自己的个人分支（如 feature/detection-张三-装甲板识别算法）。

  - 目标分支（Base）：选择自己组的开发分支（识别组选 dev/detection，解算组选 dev/solve，通信组选 dev/communication）。

4. 点击「Create pull request」，进入PR编辑页面。

5. 填写PR标题和描述：
        

  - 标题：遵循「组别-姓名-功能」格式（如：识别组-张三-装甲板识别算法开发完成）。

  - 描述：简要说明开发的功能、测试情况、是否有需要注意的问题（如：已完成装甲板识别基础算法，测试通过，无明显bug）。

6. 在「Assignees」（负责人）中，选择本组的负责人（联系负责人确认其GitHub账号），点击「Create pull request」，完成验收申请提交。

注意：提交PR后，需及时通知本组负责人进行验收；验收未通过时，根据负责人的反馈修改代码，修改完成后重新提交（无需重新创建PR，推送修改后的代码到个人分支，PR会自动更新）。

6. 验收通过后，同步最新代码（可选但推荐）

当负责人验收通过，将你的个人分支合并到本组开发分支后，建议你同步本组开发分支的最新代码，便于后续开发：

# 切换到本组开发分支
git checkout dev/detection  # 识别组，其他组替换为对应分支
# 拉取最新代码
git pull
# 切换回自己的个人分支（若需继续开发）
git checkout 个人分支名

三、各组负责人操作流程

1. 日常维护（每日必做）

1. 切换到本组开发分支，拉取最新代码，确保自己本地的分支与远程同步：
        git checkout dev/detection  # 识别组，其他组替换为对应分支
git pull

2. 关注本组组员提交的PR，及时查看、验收（建议每日至少查看1次）。

2. 代码验收与合并（组员提交PR后）

1. 收到组员的PR通知后，打开GitHub仓库的「Pull requests」页面，找到对应组员的PR。

2. 验收代码：


  - 查看代码是否符合开发规范（命名、格式、功能实现）。

  - 测试代码是否能正常运行，是否存在bug、语法错误。

  - 若代码存在问题，在PR下方的「Conversation」中留言，告知组员修改方向，驳回PR。

  - 若代码无问题，点击PR页面中的「Merge pull request」→「Confirm merge」，将组员的个人分支合并到本组开发分支。

3. 合并完成后，删除组员的个人分支（可选，建议删除，保持仓库分支整洁）：点击「Delete branch」。

3. 申请合并到主分支（本组功能阶段性完成后）

当本组开发分支上的功能完成一个阶段（如一个完整模块开发完成、测试稳定），可申请将本组开发分支合并到 main 主分支，步骤如下：

1. 确保本组开发分支代码稳定、测试通过，无明显bug。

2. 在GitHub上创建PR：
        

  - 源分支（Compare）：本组开发分支（如 dev/detection）。

  - 目标分支（Base）：main 主分支。

3. 填写PR标题和描述，说明合并的功能、测试情况，指派最终负责人（战队技术组长）进行审核。

4. 等待最终负责人审核通过后，完成合并（仅最终负责人有权合并到main分支）。

四、常见问题与解决方案（新手必看）

1. 克隆仓库失败

问题表现：终端提示“fatal: could not read from remote repository”。

解决方案：

- 检查GitHub账号是否登录，是否有仓库访问权限（联系负责人确认）。

- 检查仓库地址是否复制正确（不要多空格、少字符）。

- 若使用HTTPS地址登录失败，尝试生成GitHub个人访问令牌（PAT），登录时用令牌代替密码（生成步骤见附录）。

2. 切换分支失败

问题表现：终端提示“error: Your local changes would be overwritten by checkout”。

解决方案：

- 若当前修改的代码无需保留，执行：git checkout -- .（放弃本地修改），再切换分支。

- 若当前修改的代码需要保留，先提交本地修改（git add . + git commit -m "临时提交"），再切换分支。

3. 推送代码失败

问题表现：终端提示“error: failed to push some refs to 'https://github.com/xxx'”。

解决方案：

- 先拉取远程对应分支的最新代码，解决冲突后再推送：git pull origin 个人分支名。

- 若提示“fatal: remote origin already exists”，说明远程仓库关联错误，执行：git remote remove origin，再重新关联仓库（git remote add origin 仓库地址）。

4. 代码冲突

问题表现：拉取、合并代码时，终端提示“Automatic merge failed; fix conflicts and then commit the result”，VSCode中代码出现标红。

解决方案：

- 在VSCode中，标红部分即为冲突代码，点击标红区域的「接受当前更改」「接受传入更改」「两者都保留」，手动解决冲突（建议与相关组员沟通，确认保留哪部分代码）。

- 冲突解决后，执行：git add . → git commit -m "fix: 解决代码冲突" → git push。

5. 提交信息不规范，被负责人驳回

解决方案：修改提交信息，重新提交并推送，步骤如下：

# 修改最近一次提交信息
git commit --amend
# 输入新的规范提交信息，保存退出
# 重新推送（若已推送过，需加 --force，谨慎使用）
git push --force origin 个人分支名

五、规范与禁忌（强制遵守）

1. 强制规范

- 分支命名、提交信息必须严格遵循本说明，否则负责人有权拒绝验收、合并。

- 所有开发工作必须在个人分支上进行，禁止直接在 main、dev/detection、dev/solve、dev/communication 分支上修改、推送代码。

- 开发前必须拉取本组开发分支的最新代码，避免代码冲突。

- 禁止提交无用文件（如 .vscode 配置文件、编译产物、缓存文件等），.gitignore 已过滤此类文件，不要修改 .gitignore。

- 代码格式整洁，命名规范（变量、函数、类名清晰，避免乱命名），便于后续维护和验收。

2. 绝对禁忌

- 禁止删除、修改他人代码（若需修改，需与相关同学、负责人沟通确认）。

- 禁止强行推送代码到 main 分支、本组开发分支（违规将取消仓库操作权限）。

- 禁止提交恶意代码、无效代码、测试垃圾代码。

- 禁止随意删除仓库文件、目录（删除需经负责人同意）。

六、附录：GitHub 个人访问令牌（PAT）生成步骤

当使用HTTPS地址推送代码，提示密码登录失败时，需生成个人访问令牌，步骤如下：

1. 打开GitHub，登录个人账号，点击右上角头像 →「Settings」。

2. 在左侧菜单中，找到「Developer settings」→「Personal access tokens」→「Tokens (classic)」。

3. 点击「Generate new token」，填写相关信息：
        

  - Note：填写令牌名称（如 HNU_NHS_Vision）。

  - Expiration：选择令牌有效期（建议选择“No expiration”，避免频繁生成）。

  - Scopes：勾选「repo」（所有repo相关权限），其他无需勾选。

4. 点击「Generate token」，生成令牌（注意：令牌仅显示一次，复制保存好，不要泄露）。

5. 推送代码时，提示输入密码，粘贴保存的令牌即可完成登录。
